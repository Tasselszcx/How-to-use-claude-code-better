## 前言

大多数人使用 Claude Code 的方式是：提一个需求 → 等它执行完 → 看结果。但这种方式在需求模糊或有多种实现路径时，往往会产生「做了半天但方向不对」的问题。

`AskUserQuestion` 是 Claude Code 内置的一个工具，它允许 Claude Code 在执行前主动向你提问，用结构化的选项收集你的决策，而不是自己猜测或直接动手。

理解并善用这个工具，能让 Claude Code 的执行准确率大幅提升。

## 一、AskUserQuestion 是什么

`AskUserQuestion` 是 Claude Code 的一个内置工具，作用是：**在执行任务前，以结构化的方式向用户提问，收集必要信息后再行动**。

它的输出形式是一个带选项的交互式问题，你选择后 Claude Code 才继续执行。

典型场景：

```
你：帮我重构这个模块

Claude Code：[AskUserQuestion]
  重构的目标是什么？
  A. 提升性能（减少不必要的计算）
  B. 提升可读性（拆分函数、改善命名）
  C. 减少代码量（合并重复逻辑）
  D. 都要，按优先级排序

你：B

Claude Code：[开始执行，专注于可读性重构]
```

---

## 二、为什么它很重要

Claude Code 面对模糊需求时，有两种策略：

1. **自己猜**：选一种它认为合理的方案直接执行，做完你才发现方向不对
2. **先问再做**：用 AskUserQuestion 收集关键信息，确认后再执行

策略 1 看起来更快，但需求越复杂，猜错的代价越高。策略 2 多花 10 秒确认，能省去大量返工时间。

> **核心价值**：把「猜测执行 → 发现错误 → 重新执行」的循环，变成「确认方向 → 一次执行到位」。

## 三、如何触发 AskUserQuestion

### 方式一：让 Claude Code 自己判断

对于复杂任务，Claude Code 会自动判断是否需要先提问。你可以在需求里加一句提示：

```
帮我设计用户权限系统。在开始之前，先问我几个关键问题，确认方向后再动手。
```

### 方式二：在 CLAUDE.md 中设置全局习惯

在 `~/.claude/CLAUDE.md` 或项目的 `CLAUDE.md` 中加入：

```markdown
## 工作方式
- 对于非平凡的任务，先用 AskUserQuestion 确认关键决策点，再开始执行
- 如果一个任务有多种合理的实现路径，必须先提问
- 不要在没有确认的情况下修改超过 3 个文件
```

### 方式三：在提示词中明确要求

```
在 plan 模式下，帮我规划这个功能的实现方案。
遇到任何不确定的地方，用 AskUserQuestion 问我，不要自己假设。
```

## 四、最佳实践

### 1. 在 plan 模式下配合使用

`plan` 模式（按 `Tab` 切换到 plan，或启动时加 `--permission-mode plan`）下，Claude Code 只规划不执行。配合 AskUserQuestion，可以在执行前完整对齐方案：

```
# 先按 Tab 把权限模式切换到 plan，再发消息
帮我规划一下登录功能的改造，有不确定的地方先问我
```

### 2. 复杂任务拆分时使用

当任务涉及多个子任务，且子任务之间有依赖关系时，在关键节点确认：

```
帮我完成这个 feature，每完成一个主要步骤后，
用 AskUserQuestion 确认下一步方向再继续
```

### 3. 有风险操作前使用

对于不可逆的操作（删除文件、修改数据库、重命名大量文件），在 CLAUDE.md 中加入：

```markdown
- 删除文件或目录前，必须用 AskUserQuestion 确认
- 修改超过 5 个文件的操作，先用 AskUserQuestion 展示计划
```

### 4. 选项设计要有实质区别

AskUserQuestion 的选项应该代表真正不同的方向，而不是「是/否」这样的简单确认：

```
# ✅ 好的提问方式（选项有实质区别）
API 错误处理策略？
A. 统一在中间件处理，返回标准错误格式
B. 在各个接口单独处理，灵活定制
C. 两种结合：中间件兜底 + 接口自定义

# ❌ 不好的提问方式（太简单，不如直接执行）
要继续吗？
A. 是
B. 否
```

## 五、实际示例

### 示例一：技术选型

```
我需要给这个 Node.js 项目加一个任务队列，帮我实现。
先问我几个问题再动手。

Claude Code 的提问：
任务队列的使用场景是什么？
A. 发送邮件/短信等轻量异步任务（推荐 Bull/BullMQ）
B. 大量数据处理，需要分布式（推荐 RabbitMQ/Kafka）
C. 简单的延迟执行，不需要持久化（推荐内存队列）
D. 告诉我你的具体场景，我来推荐
```

### 示例二：重构策略

```
帮我重构 src/api/ 目录下的接口层代码

Claude Code 的提问：
重构的主要目标？（可多选）
A. 统一错误处理
B. 提取重复的参数验证逻辑
C. 改善代码组织结构（按功能模块拆分）
D. 添加 TypeScript 类型
```

### 示例三：测试策略

```
为 UserService 添加测试

Claude Code 的提问：
测试覆盖范围？
A. 只测 happy path（核心功能正常运行）
B. 包含边界条件和错误场景
C. 全量覆盖，包含集成测试
```

## 六、总结：什么时候用

| 场景 | 是否使用 AskUserQuestion |
|------|------------------------|
| 明确的单一任务（"修复这个 bug"） | ❌ 不需要，直接执行 |
| 有多种合理实现路径 | ✅ 需要，先确认方向 |
| 涉及架构或技术选型 | ✅ 需要，影响范围大 |
| 不可逆操作（删除、重命名） | ✅ 需要，避免误操作 |
| 需求描述模糊 | ✅ 需要，澄清后再执行 |
| 任务涉及多个文件/模块 | ✅ 建议先确认计划 |

> **核心原则**：让 Claude Code 在执行前把不确定性消除掉，而不是执行后再来纠正。
