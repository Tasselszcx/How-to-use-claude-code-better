## 前言

代码审查是 Claude Code 最成熟的使用场景之一。它不会因为疲劳而漏掉问题，不会因为熟悉代码而产生盲点，也不会因为碍于情面而不指出问题。

本文介绍如何用 Claude Code 做高质量的代码审查，包括内置的 `/review` 命令、自定义审查规则，以及如何融入团队的 git workflow。

## 一、最快的方式：/review 命令

在当前分支有改动时，直接执行：

```
/review
```

Claude Code 会自动：
1. 读取当前分支与主分支的 diff
2. 理解改动的上下文（相关文件、函数用途）
3. 给出审查意见

适合快速检查自己的改动，提交前自审。

---

## 二、定向审查：指定关注点

`/review` 的默认审查比较全面，但有时你只关心某个维度：

```
审查 @src/auth/jwt.ts 的改动，只关注安全性：
- JWT token 的生成和验证是否有漏洞
- 过期时间处理是否正确
- 是否有 token 泄漏的风险
不需要评论代码风格
```

```
帮我审查这次提交（git diff HEAD~1），
重点看性能影响：
- 有没有引入 N+1 查询
- 循环里有没有不必要的数据库调用
- 缓存使用是否合理
```

## 三、审查特定文件或 PR

### 审查本地改动

```bash
# 先在 Claude Code 里执行，查看暂存区改动
! git diff --staged
```

```
# 再发消息让它审查
帮我审查上面这些改动，提交前检查一下
```

```
# 或直接引用某次提交范围
帮我审查 @git:HEAD~1..HEAD 的改动
```

### 审查 Pull Request

```
帮我审查 PR #123，
重点关注：
1. 业务逻辑是否正确（对照 @docs/spec.md 里的需求）
2. 错误处理是否完整
3. 有没有遗漏的测试场景
```

## 四、在 CLAUDE.md 中配置审查规范

把团队的代码审查标准写进 CLAUDE.md，每次审查都会自动遵循：

```markdown
## 代码审查规范

### 必须检查
- SQL 注入、XSS 等安全漏洞
- 空指针/空值未处理
- 错误没有被捕获或记录
- 敏感信息（密码、token）出现在日志或响应中

### 性能检查
- 循环内的数据库查询（N+1 问题）
- 大数据量未分页
- 未使用索引的查询条件

### 代码规范
- 函数超过 50 行需要拆分
- 重复代码超过 3 次需要提取
- 变量命名要能自解释

### 审查输出格式
按严重程度分级：
- 🔴 必须修改（安全问题、逻辑错误）
- 🟡 建议修改（性能问题、可读性）
- 🟢 可选优化（代码风格、小改进）
```

## 五、多维度并行审查（Teams 模式）

对于重要的 PR，可以用 Teams 模式同时从多个维度审查：

```
对 @src/payment/ 目录的改动做全面审查。
创建 3 个审查员 Agent 并行工作：

- security-reviewer：专注安全性
  重点：支付金额篡改、重放攻击、权限绕过

- logic-reviewer：专注业务逻辑
  重点：对照 @docs/payment-spec.md，验证各种支付场景是否正确处理

- quality-reviewer：专注代码质量
  重点：错误处理完整性、测试覆盖、可维护性

每个审查员独立出报告，最后汇总。
```

## 六、审查后自动修复

发现问题后，可以让 Claude Code 直接修复：

```
根据你的审查意见，帮我修复所有"必须修改"级别的问题。
修改前对每个问题说明修复方案，等我确认后再动手。
```

或者针对某个具体问题：

```
你提到第 47 行有 SQL 注入风险，帮我修复这个问题，
使用参数化查询替换字符串拼接
```

## 七、集成到 git workflow

### 提交前自动审查（git hook）

在 `.git/hooks/pre-push` 中加入：

```bash
#!/bin/bash
echo "正在进行 Claude Code 代码审查..."
claude -p "审查 git diff origin/main...HEAD 的改动，
如果发现严重问题（安全漏洞、明显的逻辑错误）输出 BLOCK，
否则输出 PASS，然后列出发现的问题" \
--dangerously-skip-permissions

# 根据输出决定是否阻止 push（可选）
```

### 在 Claude Code Hook 中配置

更优雅的方式是通过 Claude Code 的 Hooks 系统，在每次任务完成后提示审查：

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "echo '提示：提交前记得用 /review 检查一下改动'"
          }
        ]
      }
    ]
  }
}
```

## 八、审查效果最大化的技巧

### 提供足够的背景

```
# ❌ 不好：没有上下文
帮我审查这个文件

# ✅ 好：提供背景和关注点
帮我审查 @src/order/create.ts，这是下单接口的核心逻辑。
背景：上周生产环境出现了重复下单的问题，已经修复了，
这次改动是加强防重复的逻辑。
重点检查：并发场景下的幂等性是否真的保证了
```

### 区分「审查」和「改进」

```
# 第一步：只审查，不修改
帮我审查 @src/utils/parser.ts，列出所有问题，不要修改代码

# 确认问题后，第二步：修复
好的，针对你列出的问题1和问题3，帮我修复
```

### 要求给出修复建议

```
审查 @src/api/user.ts，对每个发现的问题，
除了说明问题在哪，还要给出具体的修复代码示例
```
