## 前言

Claude Code 的效果好不好，提问方式占了一大半。同样的任务，「帮我修一下这个 bug」和「这个函数在处理空数组时会崩溃，报错是 `TypeError: Cannot read property 'length' of undefined`，帮我修复并加上防御性检查」，得到的结果天差地别。

本文不讲「如何写好 prompt」的理论，只讲在 Claude Code 这个具体场景下哪些做法真的有效。

## 一、给足上下文，而不是让它猜

Claude Code 启动时会扫描当前目录，但它不会主动读取所有文件。你需要主动告诉它「去哪里看」。

### 用 @ 引用具体文件

```
# ❌ 不好：让 Claude Code 自己去猜
帮我优化登录逻辑

# ✅ 好：直接告诉它看哪里
帮我优化 @src/auth/login.ts 里的登录逻辑，
主要问题是每次登录都会重新创建数据库连接
```

### 描述问题的背景

```
# ❌ 不好
这个接口很慢，帮我优化

# ✅ 好
@src/api/user.ts 里的 getUserList 接口，
在用户量超过 1 万时响应时间超过 3 秒。
数据库用的是 MySQL，已经有 user_id 的索引。
帮我分析瓶颈在哪，给出优化方案
```

### 告诉它你已经知道的

避免让 Claude Code 重复你已经做过的排查：

```
我已经确认了：
1. 网络请求正常，接口能收到请求
2. 数据库查询很快（< 50ms）
3. 问题出在数据处理阶段

请帮我看 @src/processor/data.ts，找出为什么处理 10 万条数据需要 5 秒
```

## 二、任务拆解：大任务拆成小步骤

一次性给一个超大任务，Claude Code 容易跑偏或做到一半卡住。

### 把大任务分阶段

```
# ❌ 不好（一次性抛出所有要求）
帮我重构整个用户模块，包括登录、注册、权限管理、
用户信息展示，要加上单元测试，还要更新文档

# ✅ 好（分阶段）
第一步：先帮我梳理 @src/user/ 目录下的现有代码结构，
列出主要的函数和它们的职责，不要修改任何代码
```

然后根据梳理结果，再下一步：

```
根据你的梳理，先重构登录相关的部分：
@src/user/login.ts 和 @src/user/auth.ts
重构目标：提取重复的验证逻辑，不改变外部接口
```

### 先规划再执行

对于复杂任务，先让它在 plan 模式下给出方案：

```
# 先切到 plan 模式（Tab 键切换，或启动时加 --permission-mode plan）
帮我设计一个用户权限系统的改造方案，
当前系统用的是简单的 role 字段，
需要改成支持细粒度权限控制（资源 + 操作）
```

确认方案后再切回执行模式。

## 三、明确期望结果，而不是描述过程

告诉 Claude Code 你要什么，而不是告诉它怎么做。

```
# ❌ 描述过程（容易产生歧义）
把这个函数改成异步的，然后加上 try-catch，
再把错误信息格式化一下

# ✅ 描述结果（更清晰）
@src/utils/fetch.ts 里的 getData 函数需要：
1. 支持异步调用，返回 Promise
2. 网络错误时返回 { error: '网络错误', code: 'NETWORK_ERROR' }
3. 超时（>5s）时返回 { error: '请求超时', code: 'TIMEOUT' }
保持现有的调用方不需要改动
```

## 四、利用 CLAUDE.md 避免重复说明

每次都要说「用 pnpm 不要用 npm」、「函数要加类型注解」很烦。把这些写进 CLAUDE.md，一劳永逸：

```markdown
# 项目规范（放在项目根目录的 CLAUDE.md）

## 必须遵守
- 包管理只用 pnpm，不要用 npm 或 yarn
- 所有函数必须有 TypeScript 类型注解
- 修改代码前先说明方案，不要直接动手
- 不引入新依赖，除非我明确同意

## 测试
- 运行测试：pnpm test
- 单个文件测试：pnpm test src/xxx.test.ts

## 禁止操作
- 不修改 .env 文件
- 不修改 prisma/schema.prisma（除非我明确要求）
```

> 关于 CLAUDE.md 的完整用法，见 [08 · CLAUDE.md 实战](./08_claude_md_in_practice.md)。

## 五、迭代式对话：把 Claude Code 当协作者

不要期望一次对话解决所有问题，要像跟同事协作一样迭代。

### 先让它理解，再让它做

```
第一轮：帮我读一下 @src/payment/ 目录，
理解当前的支付流程，用几句话描述一下，不要修改代码

第二轮：（确认它理解正确后）
好的，现在帮我在支付成功后增加一个发送短信通知的步骤，
短信服务的接口在 @src/services/sms.ts
```

### 对结果给出明确反馈

```
# ❌ 模糊反馈（Claude Code 不知道哪里不对）
不对，重新做

# ✅ 明确反馈
你的修改逻辑是对的，但有两个问题：
1. 第 23 行的 userId 应该是 number 类型，不是 string
2. 错误处理只覆盖了网络错误，还需要处理 404 的情况
```

### 用 /clear 开启新任务

完成一个任务后，用 `/clear` 清空上下文再开始下一个，避免旧任务的上下文干扰新任务。

## 六、常见低效模式和改进方式

| 低效做法 | 问题 | 改进方式 |
|---------|------|----------|
| 「帮我修一下这个 bug」 | 没有描述 bug 是什么 | 描述报错信息、复现步骤、期望行为 |
| 「优化一下代码」 | 优化方向不明确 | 说明是性能、可读性还是代码量 |
| 「按最佳实践来」 | 最佳实践因项目而异 | 在 CLAUDE.md 中写明项目规范 |
| 一次提交十个需求 | 容易遗漏或做错 | 一次一个任务，完成后再下一个 |
| 不引用具体文件 | Claude Code 需要猜测 | 用 @ 明确指定相关文件 |
| 对结果说「不对」 | 无法定向修正 | 具体说明哪里不对、期望是什么 |

## 七、几个实用的提问模板

### 修复 Bug

```
@文件路径 里的 函数名 函数报错：
错误信息：[粘贴完整报错]
触发条件：[什么情况下会出现]
期望行为：[应该怎样]
已排查：[你已经确认的信息]
```

### 新增功能

```
需要在 @文件路径 里新增 功能描述。
输入：[参数说明]
输出：[返回值说明]
边界情况：[需要处理的特殊情况]
不要修改：[不能动的部分]
```

### 代码审查

```
请审查 @文件路径，重点关注：
1. [关注点1，如安全性]
2. [关注点2，如性能]
3. [关注点3，如错误处理]
不需要修改代码，只给出审查意见
```

### 理解代码

```
帮我理解 @文件路径 里的 函数名 函数：
1. 它的主要逻辑是什么
2. 为什么要这样写（有没有特殊考虑）
3. 有哪些潜在的问题
不要修改任何代码
```
